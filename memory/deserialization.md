# 反序列化题型要点

## 在本项目中的作用
- 这是题型识别与代码审计提示，不是完整利用手册。
- 当题目被识别为 `deserialization`，应优先把它当成“找触发点 + 找可控对象流 + 找危险 sink”的代码分析任务。

## 优先关注的触发点
- PHP：`unserialize()`、Phar 相关文件函数链
- Python：`pickle.loads()`、不可信反序列化入口
- Java：`ObjectInputStream.readObject()` 一类对象反序列化入口

## 代码审计重点
- 输入是否可控：Cookie、POST body、session、上传文件、数据库回填值
- 是否存在对象图或 gadget chain：魔术方法、回调、动态调用、文件操作、命令执行、模板渲染
- 反序列化后是否进入危险行为：执行命令、包含文件、写文件、访问任意路径、发起请求
- 是否存在可用于探测的副作用：报错、日志、DNS/HTTP 请求、文件落地、权限变化

## PHP 方向高价值线索
- 魔术方法：`__wakeup()`、`__destruct()`、`__toString()`、`__get()`、`__call()`
- 常见危险点：`eval`、`system`、文件写入/包含、可控回调
- 若题目包含文件上传、图片处理、压缩包处理，要额外考虑 Phar 元数据触发链

## Python / Java 方向高价值线索
- Python：看 `pickle`、自定义 `__reduce__` 相关对象流、插件或缓存反序列化逻辑
- Java：看常见第三方链依赖、反序列化入口是否暴露给网络输入、是否有明显回显或外带信号

## 解题流程提醒
1. 先确认反序列化入口是否可达。
2. 再确认输入是否真正可控。
3. 优先做源码审计，找类关系与危险 sink。
4. 没有足够证据时不要盲猜利用链。
5. 若常规方向失败，回到项目主链继续：记录证据、更新短期记忆、必要时求助。

## 与项目主链的关系
- 反序列化题通常更依赖 `source_analysis` 与代码理解，不应只靠盲打工具。
- 若用户提供源码，应优先把源码分析结果纳入 planner 输入。
